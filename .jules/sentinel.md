## 2024-05-04 - PII Exposure via Insecure Direct Object Reference and Verbose Error Handling
**Vulnerability:** The `GET /api/requests` endpoint was returning the patient's phone number to anyone who queried it, allowing nurses to see patient contact info before accepting a request. Additionally, the endpoint exposed raw database error messages (`err.message`) in the response upon failure.
**Learning:** In the absence of robust server-side authorization or filtering logic, endpoints that return data from joined tables easily leak Personally Identifiable Information (PII) if the fields are not strictly filtered at the database query level. Furthermore, defaulting to `res.json({ error: err.message })` exposes internal database structures and logic.
**Prevention:** To prevent excessive data exposure, conditionally mask PII directly via database queries (e.g., `CASE WHEN r.status = 'confirmed' THEN u.phone ELSE NULL END as patientPhone`). For error handling, catch database errors, log them server-side via `console.error(err)`, and return a generic error message (e.g., `{ "error": "Internal server error" }`) to the client.
